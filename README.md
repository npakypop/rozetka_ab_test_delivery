# Full-Stack аналіз A/B-тесту та юніт-економіки: Інтеграція GA4 + KeyCRM

## 📌 Огляд проєкту
Цей репозиторій містить опис наскрізного пайплайну даних, дизайну експерименту та статистичної оцінки результатів A/B-тесту для e-commerce. 

**Бізнес-проблема:** Команда маркетингу висунула гіпотезу, що додавання банера **"Безкоштовна доставка"** на сторінках товарів стимулюватиме користувачів завершувати покупки, що збільшить коефіцієнт конверсії та загальний дохід. 

**Аналітичний підхід:** Замість того, щоб сліпо покладатися на поверхневі метрики веб-аналітики (кліки або нестабільні фронтенд-конверсії), у цьому проєкті було побудовано уніфікований full-stack пайплайн даних. Шляхом об'єднання поведінкових даних клікстріму з фронтенду (**Google Analytics 4**) із транзакційними даними з бекенду (**KeyCRM**), аналіз дозволив оцінити реальний вплив фічі на **чистий прибуток, юніт-економіку та фактичні логістичні витрати**.

## ⚙️ Інфраструктура даних та налаштування тесту

Щоб гарантувати повну цілісність даних та усунути проблеми з видаленням файлів cookie чи зміною груп користувачами, інфраструктуру трекінгу було спроєктовано з нуля:
* **Пайплайн даних CRM:** Автоматизовано щоденну синхронізацію деталізованих логів замовлень із **KeyCRM до Google BigQuery**, включаючи критично важливі змінні бекенду: статуси замовлень, маржинальність, дохід від доставки та фактичні витрати на кур'єрів/доставку.
* **Рандомізація трафіку через GTM:** У Google Tag Manager було впроваджено кастомний скрипт розподілу користувачів на JavaScript. Скрипт використовував **first-party cookies**, щоб закріпити користувачів за їхньою початковою когортою (`hidden_banner` проти `visible_banner`) під час першої сесії. Це забезпечило сувору ізоляцію експерименту протягом багатоденних повторних візитів.
* **Синхронізація когорт:** Призначений варіант передавався безпосередньо у клікстрім GA4 як постійна **User Property** (`ab_test_rozetka_variant`), пов'язуючи всі наступні події з реальною когортою користувача.

## 📈 Результати A/B-тесту та бізнес-інсайти

### 1. Перевірка якості: Sample Ratio Mismatch (SRM)
Щоб переконатися, що механізм рандомізації розділив трафік точно 50/50, було проведено тест відповідності Хі-квадрат (Chi-Square goodness-of-fit):  
* **Результат P-value:** 0.7733  
* **Висновок:** SRM не виявлено. Розподіл розміру вибірки між `hidden_banner` та `visible_banner` є статистично рівномірним, що дозволяє проводити подальшу оцінку.  

### 2. Основний аналіз: Intent-to-Treat (ITT) проти Exposed воронки
Стандартний, поверхневий аналіз для всіх відвідувачів сайту показав майже непомітне падіння конверсії на 0.1%.  

Проте, оскільки банер фізично знаходився всередині сторінки товару, користувачі, які купували без переходу на сторінку картки товару (наприклад, швидкі покупки з категорійної сітки), створювали сильний аналітичний шум. Фільтрація датасету для оцінки лише тих користувачів, які взаємодіяли з банером (`user_item_views > 0`), показала зовсім іншу картину:  

| Метрика | Варіант A (Прихований банер) | Варіант B (Видимий банер) | Абсолютна зміна | Відносна зміна |
| :--- | :---: | :---: | :---: | :---: |
| **Усього користувачів** | 3,652 | 3,753 | — | — |
| **Конвертовані користувачі** | 252 | 241 | — | — |
| **Коефіцієнт конверсії (CR)** | 6.90% | 6.42% | -0.48% в.п. | -7.24% |

Z-тест показав p-value = 0.4086. Хоча цей показник не перетинає суворий поріг значущості у 95% через обмежений розмір локальної вибірки, стабільне зниження метрик на кожному кроці мікроконверсії змусило відхилити позитивну продуктову гіпотезу.  

### 3. Фінансовий вплив та юніт-економіка
Завдяки підтягуванню бекенд-логів із KeyCRM, метрики чітко підсвітили фінансові наслідки цієї зміни:
* Банер відволікав користувачів та створював додаткове когнітивне навантаження, знижуючи загальну частоту покупок.  
* Інтеграція змінної `logistics_ratio` довела, що масштабування цієї фічі на весь проєкт призвело б до стискання маржі чистого прибутку компанії через приховані операційні витрати на обробку замовлень, без жодного приросту в обсягах продажу.

## 💡 Фінальний вердикт та рекомендації

* **Скасування впровадження (Rollback):** Наполегливо рекомендовано продуктовому менеджменту не впроваджувати банер "Безкоштовна доставка" на сторінках товарів на постійній основі. Його запуск погіршить показники конверсії та знизить ефективність валової маржі.  
* **Оптимізація UX-тертя:** Експеримент довів, що згадка умов безкоштовної доставки безпосередньо на детальній сторінці товару працює швидше як когнітивний відволікаючий фактор, ніж як стимул.
* **Альтернативна стратегія:** Рекомендовано перенести стимул "Безкоштовна доставка" далі за воронкою оформлення (наприклад, інтегрувати його в Кошик як тригер апсейлу для збільшення середнього чека — AOV).
